Performance Simulation Details

SessionsSimulation.scala

Load test for the session retrieval API. This endpoint is used by AI QA Lab, Research Dashboard, Admin Panel and session selectors.

Endpoint GET /api/sessions
Purpose Validate session list loading and database read performance.
Load model Ramp users + constant requests per second.
Gate P95 < 500 ms and success rate > 95%
PASS means The platform can load saved sessions quickly under concurrent access.

DuringMergeSimulation.scala

Performance test for telemetry synchronization. It validates the DURING phase merge engine used to align Garmin FIT and pulseox CSV data into a single timeline.

Endpoint POST /api/during_merge
Purpose Validate merge latency and pipeline stability.
Signals HR, HRV, RR interval, pulse and SpO₂.
Gate P95 < 1000 ms and failed requests < 10%
PASS means The merge engine can process telemetry timelines without performance degradation.

RunAnalysisSimulation.scala

Performance benchmark for the AI analysis pipeline. It validates score generation, anomaly detection, physiological features, risk level, timeline and research disclaimer.

Endpoint POST /api/run_analysis
Purpose Validate AI scoring latency and response contract stability.
Validated fields score, risk_level, anomaly, features, timeline.
Gate P95 < 1500 ms and success rate > 95%
PASS means The AI engine can respond fast enough for dashboard and QA workflows.

UploadCsvSimulation.scala

Upload and parser performance test for pulseox CSV telemetry. It validates ingestion speed, parser stability and database insert behavior using realistic multipart CSV payloads with pulse and SpO₂ samples.

Endpoint POST /upload_csv
Purpose Validate telemetry ingestion and CSV parser throughput.
Payload Multipart CSV file with pulse and SpO₂ samples.
Validated now Multipart upload, CSV parser readiness, response latency and controlled failure rate.
Gate P95 < 2000 ms and failed requests < 20%
Target gate P95 < 2000 ms and success rate > 95%
PASS means The platform can ingest pulseox telemetry reliably.

UploadFitSimulation.scala

Upload and parser performance test for Garmin FIT telemetry. It validates FIT ingestion latency, upload pipeline stability, HR extraction readiness, RR interval parsing readiness and HRV pipeline readiness.

Endpoint POST /upload_fit
Purpose Validate Garmin FIT ingestion and parser performance.
Payload Multipart FIT-like file payload for endpoint and parser-path validation.
Signals HR, RR intervals and HRV-ready telemetry path.
Validated now Multipart upload handling, parser-path responsiveness and controlled parser failure behavior.
Gate P95 < 2500 ms and parser failure rate controlled.
Target gate P95 < 2500 ms and success rate > 95% with real FIT fixture.
PASS means The platform can ingest Garmin FIT telemetry reliably.

Performance Gates

Area Target Gate Meaning
Sessions API < 300 ms preferred / < 500 ms gate PASS / FAIL Dashboard and selector loading should stay fast.
Merge Engine < 1000 ms PASS / FAIL Telemetry alignment should not block the workflow.
AI Analysis < 1500 ms PASS / FAIL AI score and anomaly response should remain interactive.
CSV Upload < 2000 ms PASS / FAIL Telemetry ingestion should remain stable.
FIT Upload < 2500 ms PASS / FAIL Garmin telemetry ingestion should remain stable.

Example Gatling Scenarios

Scenario: Sessions API performance

Given saved HBOT sessions exist
When multiple virtual users call /api/sessions
Then 95% of responses should complete below 500 ms
And success rate should remain above 95%


Scenario: AI analysis performance

Given a saved HBOT session exists
When 50 virtual users call /api/run_analysis
Then 95% of responses should complete below 1500 ms
And error rate should remain below 1%


Scenario: Merge engine performance

Given FIT and CSV telemetry data exist for a session
When virtual users call /api/during_merge
Then the unified timeline should be generated within the performance gate


Scenario: CSV ingestion performance

Given a valid pulseox CSV file exists
When virtual users upload telemetry through /upload_csv
Then parser response should remain stable
And no CSV parser failures should occur


Scenario: FIT ingestion performance

Given a valid Garmin FIT file exists
When virtual users upload telemetry through /upload_fit
Then HR and RR intervals should be extracted successfully
And parser response time should remain within the performance gate
And parser failure rate should remain controlled

CI/CD Integration

Gatling simulations can be executed as a separate GitHub Actions job after the Playwright QA job. This allows the project to validate both functional correctness and performance regressions. 

CI/CD Flow

1. Checkout repository
2. Install Python dependencies
3. Install Node + Playwright
4. Start Flask backend
5. Run Playwright QA
6. Run Gatling performance simulations
7. Upload Playwright and Gatling reports
8. Fail pipeline if performance gates are violated
